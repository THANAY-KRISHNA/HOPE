Important Formulas and Theorems

CAP Theorem
Distributed system can guarantee at most 2 of 3:
Consistency, Availability, Partition Tolerance
  • CA: Traditional RDBMS (no partition tolerance)
  • CP: MongoDB, HBase (sacrifice availability)
  • AP: Cassandra, CouchDB (sacrifice consistency)
    • Quorum for Strong Consistency
    R + W > N
    where R = read replicas, W = write replicas, N = total replicas
    Example:

    N = 3, R = 2, W = 2 → Strong consistency guaranteed
    2 + 2 > 3 → 4 > 3 ✓

    Normalization Rules
    2NF: No partial dependencies (non-key attributes depend on full PK)
    3NF: No transitive dependencies (non-key attributes depend only on PK)
    BCNF: Every determinant is a candidate key
    Transaction Serializability
    Conflict Serializable: If precedence graph is acyclic
    Recoverable: Tj commits only after all Ti it read from commit
    Cascadeless: No transaction reads uncommitted data
    Strict: No transaction reads OR writes uncommitted data
    COMMON PROBLEM TYPES WITH SOLUTIONS

    Type 1: ER Diagram Design (10 marks)

    Solution Approach:
    1
    Identify entities: BOOK, MEMBER, LOAN, PUBLISHER
    2
    Identify attributes with keys: Book(ISBN, Title, Author), Member(MemberID, Name, Address)
    3
    Identify relationships with cardinalities:
    • PUBLISHER publishes BOOK (1:N)
    • MEMBER borrows BOOK through LOAN (M:N via LOAN entity)
    4
    Add constraints: Weak entities, participation constraints
    5
    Draw diagram with proper notation

    Type 2: SQL Query Writing (10 marks)

    Given tables:

    Students(id, name, dept)
    Courses(code, title, credits)
    Enrollments(student_id, course_code, semester, grade)

    Queries Required:
    1
    Find students who scored 'A' in more than 3 courses
    2
    Find department with highest average GPA
    3
    Find prerequisite chain for a given course
    Solution Approach:

    -- 1. Students with 'A' in more than 3 courses
    SELECT s.id, s.name, COUNT(*) as a_count
    FROM Students s
    JOIN Enrollments e ON s.id = e.student_id
    WHERE e.grade = 'A'
    GROUP BY s.id, s.name
    HAVING COUNT(*) > 3;

    -- 2. Department with highest average GPA
    SELECT s.dept, AVG(
    CASE e.grade
    WHEN 'A' THEN 4.0 WHEN 'B' THEN 3.0
    WHEN 'C' THEN 2.0 WHEN 'D' THEN 1.0
    ELSE 0 END) as avg_gpa
    FROM Students s
    JOIN Enrollments e ON s.id = e.student_id
    GROUP BY s.dept
    ORDER BY avg_gpa DESC
    LIMIT 1;

    -- 3. Prerequisite chain (assuming Prereqs table)
    WITH RECURSIVE prereq_chain AS (
    SELECT course_code, prereq_code, 1 as level
    FROM Prerequisites WHERE course_code = 'C101'
    UNION ALL
    SELECT p.course_code, p.prereq_code, pc.level + 1
    FROM Prerequisites p
    JOIN prereq_chain pc ON p.course_code = pc.prereq_code
    )
    SELECT * FROM prereq_chain ORDER BY level;

    Type 3: Normalization Problem (10 marks)

    Given:

    R(A, B, C, D, E) with FDs: AB → C, C → D, D → E, E → A

    Solution Steps:
    1
    Find all candidate keys: Compute closures
    2
    Find minimal cover: Remove extraneous attributes
    3
    Decompose to 3NF preserving dependencies
    4
    Decompose to BCNF (may lose dependencies)

    Type 4: Transaction Schedule Analysis (10 marks)

    Given schedule S:

    R1(A), W1(A), R2(A), W2(B), R1(B), W1(C), C1, R2(C), W2(A), C2

    Analysis Tasks:
    1
    Is it conflict serializable? Draw precedence graph
    2
    Is it recoverable? Cascadeless? Strict?
    3
    Show equivalent serial schedule if exists

    Essential Formulas & Definitions
    Closure Algorithm: X⁺ = X + {A | Y→A in F and Y ⊆ X⁺}
    Candidate Key: X is CK if X⁺ = U and no proper subset has X⁺ = U
    3NF: For X→A, either X is superkey OR A is prime
    BCNF: For X→A, X must be superkey
    CAP Theorem: Choose 2 of 3: C, A, P
    Quorum: R + W > N for strong consistency
    2PL: Growing phase (acquire locks), Shrinking phase (release locks)
    Recoverable: If Tj reads from Ti, then Ci < Cj
    Cascadeless: No transaction reads uncommitted data
    Conflict Serializable: Precedence graph is acyclic
    ACID: Atomicity, Consistency, Isolation, Durability
    BASE: Basically Available, Soft state, Eventual consistency
    SQL JOIN Types: INNER, LEFT, RIGHT, FULL, CROSS, SELF
    ER Symbols: Rectangle=Entity, Ellipse=Attribute, Diamond=Relationship
    Double Rectangle: Weak Entity, Double Diamond: Identifying Relationship
    Double Line: Total Participation, Single Line: Partial Participation
    (min,max): Cardinality notation in ER diagrams
