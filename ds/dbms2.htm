The Relational Data Model and Relational Database Constraints

Introduction to Relational Model

The Relational Model was proposed by Dr. Edgar F. Codd in 1970 in his seminal paper "A Relational Model of Data for Large Shared Data Banks." It revolutionized database systems by providing a simple, mathematical foundation for data management.

Key Innovations by Codd:

  • Data Independence: Separation of logical and physical representations
  • Set-Theoretic Foundation: Based on mathematical relations
  • Declarative Querying: Users specify what they want, not how to get it
  • Normalization: Systematic approach to eliminate data redundancy
Figure: Relational Model Architecture

Basic Concepts of Relational Model

1. Domain (D)

A domain is a set of atomic (indivisible) values. Each value in a domain is of the same data type.

Examples of Domains:
D1 = Student_IDs = {101, 102, 103, ...} // INTEGER, 3 digits D2 = Names = {"John", "Jane", "Bob", ...} // VARCHAR(50) D3 = GPAs = {0.0, 0.1, ..., 4.0} // DECIMAL(3,1), 0.0-4.0 D4 = Grades = {"A", "B", "C", "D", "F"} // CHAR(1)

2. Relation (R)

A relation is a subset of the Cartesian product of one or more domains. A relation is a set of tuples (no duplicates).

Mathematical Term Database Term
Relation Table
Tuple Row, Record
Attribute Column, Field
Cardinality Number of rows

Relational Database Constraints

Constraints are rules that must be satisfied by every valid instance of the database.

1. Domain Constraints

Each attribute value must be an atomic value from its domain.

CREATE TABLE Student ( StudentID INT CHECK (StudentID BETWEEN 100 AND 999), Name VARCHAR(50) NOT NULL, GPA DECIMAL(3,2) CHECK (GPA BETWEEN 0.0 AND 4.0), Gender CHAR(1) CHECK (Gender IN ('M', 'F', 'O')) );

2. Key Constraints

  • Superkey: A set of attributes that uniquely identifies tuples
  • Candidate Key: A minimal superkey
  • Primary Key: The chosen candidate key for identifying tuples
  • Foreign Key: Attributes that reference primary key of another relation
Figure: Key Relationships in Database

3. Entity Integrity Constraint

Rule: No primary key attribute can be NULL. Primary key identifies tuples; NULL means unknown/undefined identifier.

4. Referential Integrity Constraint

Rule: A foreign key value must either match a primary key value in the referenced relation or be NULL.

CREATE TABLE Employee ( EmpID INT PRIMARY KEY, Name VARCHAR(50), DeptID INT, FOREIGN KEY (DeptID) REFERENCES Department(DeptID) ON DELETE CASCADE -- Delete employees when department deleted ON UPDATE CASCADE -- Update DeptID in employees when changed in department );

Relational Algebra and Relational Calculus

Relational Algebra

Relational Algebra is a procedural query language that provides a set of operations to manipulate relations. It forms the theoretical foundation for SQL.

Figure: Relational Algebra Operations

Basic Operations

  • SELECT (σ): σₚ(R) - Selects tuples that satisfy predicate p
  • PROJECT (π): πₐ₁,ₐ₂,...,ₐₙ(R) - Selects columns (attributes)
  • UNION (∪): R ∪ S - Combines tuples from two relations
  • SET DIFFERENCE (-): R - S - Tuples in R but not in S
  • CARTESIAN PRODUCT (×): R × S - Combines every tuple of R with every tuple of S
Relational Algebra Examples:
-- Employees with salary > 50000 σ_(Salary > 50000)(EMPLOYEE) -- Only name and salary columns π_Name,Salary(EMPLOYEE) -- Employees in Dept 10 or Dept 20 σ_DeptID=10(EMPLOYEE) ∪ σ_DeptID=20(EMPLOYEE) -- Natural join between EMPLOYEE and DEPARTMENT EMPLOYEE ⨝* DEPARTMENT

Relational Calculus

Relational Calculus is a non-procedural (declarative) query language. Users specify what data they want without specifying how to get it.

Tuple Relational Calculus (TRC)

General Form: {t | P(t)} where t is a tuple variable and P(t) is a formula describing t.

-- Employees with salary > 50000 {t | EMPLOYEE(t) ∧ t.Salary > 50000} -- Employees who work on project 'ProjectX' {t | EMPLOYEE(t) ∧ ∃w(WORKS_ON(w) ∧ w.EmpID = t.EmpID ∧ ∃p(PROJECT(p) ∧ p.ProjID = w.ProjID ∧ p.Name = 'ProjectX'))}

Domain Relational Calculus (DRC)

General Form: {<x₁, x₂, ..., xₙ> | P(x₁, x₂, ..., xₙ)}

-- Employees with salary > 50000 { | EMPLOYEE(id, name, sal, did) ∧ sal > 50000}

Structured Query Language (SQL)

Data Definition Language (DDL)

SQL commands for defining database structure.

-- CREATE TABLE CREATE TABLE Employee ( EmpID INT PRIMARY KEY, Name VARCHAR(50) NOT NULL, Salary DECIMAL(10,2), HireDate DATE DEFAULT CURRENT_DATE, DeptID INT, FOREIGN KEY (DeptID) REFERENCES Department(DeptID) ); -- ALTER TABLE ALTER TABLE Employee ADD Email VARCHAR(100); -- DROP TABLE DROP TABLE IF EXISTS OldEmployee;

Data Manipulation Language (DML)

SQL commands for manipulating data.

SELECT Statement

-- Basic SELECT SELECT * FROM Employee; -- SELECT with WHERE SELECT Name, Salary FROM Employee WHERE Salary > 50000; -- SELECT with JOIN SELECT e.Name, d.DeptName FROM Employee e INNER JOIN Department d ON e.DeptID = d.DeptID; -- SELECT with GROUP BY SELECT DeptID, AVG(Salary) as AvgSalary FROM Employee GROUP BY DeptID HAVING AVG(Salary) > 50000;

ER-to-Relational Mapping

Systematic process to convert Entity-Relationship diagrams to relational schemas.

Figure: ER-to-Relational Mapping Process

Mapping Steps Summary

  1. Regular Entity Types: Create relation with all simple attributes
  2. Weak Entity Types: Create relation with owner's PK as FK, composite PK
  3. Binary 1:1 Relationships: Foreign key in one relation
  4. Binary 1:N Relationships: Foreign key in the "N" side
  5. Binary M:N Relationships: New relation with FKs from both entities
  6. Multi-valued Attributes: New relation with entity PK + multi-valued attribute
  7. N-ary Relationships: New relation with FKs from all participating entities
-- Example: Complete COMPANY database schema CREATE TABLE DEPARTMENT ( DeptID INT PRIMARY KEY, Name VARCHAR(50) NOT NULL, Location VARCHAR(100) ); CREATE TABLE EMPLOYEE ( EmpID INT PRIMARY KEY, Name VARCHAR(50) NOT NULL, Salary DECIMAL(10,2), BirthDate DATE, DeptID INT, SupervisorID INT, FOREIGN KEY (DeptID) REFERENCES DEPARTMENT(DeptID), FOREIGN KEY (SupervisorID) REFERENCES EMPLOYEE(EmpID) ); CREATE TABLE PROJECT ( ProjID INT PRIMARY KEY, Name VARCHAR(50) NOT NULL, Location VARCHAR(100), DeptID INT, FOREIGN KEY (DeptID) REFERENCES DEPARTMENT(DeptID) ); CREATE TABLE WORKS_ON ( EmpID INT, ProjID INT, Hours DECIMAL(5,2), PRIMARY KEY (EmpID, ProjID), FOREIGN KEY (EmpID) REFERENCES EMPLOYEE(EmpID), FOREIGN KEY (ProjID) REFERENCES PROJECT(ProjID) );
