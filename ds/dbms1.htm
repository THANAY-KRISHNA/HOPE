1.1 DATABASE SYSTEM CONCEPTS AND ARCHITECTURE

What is a Database System?

A Database System (DBS) is an integrated collection of related data along with a set of programs to access, manage, and control the data. It is designed to handle large amounts of information by storing, retrieving, and managing data efficiently.

Components of a Database System:

  1. Database (DB): Organized collection of logically related data
  2. Database Management System (DBMS): Software system that enables users to define, create, maintain, and control access to the database
  3. Database Applications: Programs that interact with the database through the DBMS
  4. Users: End users, application programmers, database administrators (DBAs)
  5. Hardware: Physical devices (computers, storage, network)
  6. Procedures: Instructions and rules for using the database system

Data Models: The Blueprint of Databases

A Data Model is a collection of concepts for describing data, data relationships, data semantics, and consistency constraints.

Categories of Data Models:

  1. Conceptual (High-level) Data Models
    • Close to how users perceive data
    • Hide physical storage details
    • Examples: Entity-Relationship (ER) model, Object-Oriented model
    • Purpose: Communication between users and designers
    • Features: Entities, attributes, relationships, constraints
  2. Physical (Low-level) Data Models
    • Describe how data is stored on computer storage media
    • Concerned with storage structures, access paths, indexing
    • Examples: File structures, B-trees, Hash indexes
    • Purpose: Database implementation and optimization
  3. Implementation (Representational) Data Models
    • Between conceptual and physical models
    • Used by many commercial DBMS implementations
    • Examples: Relational model, Network model, Hierarchical model
    • Purpose: Directly implementable in database systems

Historical Evolution of Data Models:

  • 1960s: Hierarchical Model (IBM's IMS)
    • Tree-like structure
    • Parent-child relationships
    • Limited to one-to-many relationships
  • 1970s: Network Model (CODASYL DBTG)
    • Graph structure
    • More flexible than hierarchical
    • Complex pointer structures
  • 1980s-Present: Relational Model (Edgar Codd, 1970)
    • Tables (relations) with rows and columns
    • Mathematical foundation (set theory, predicate logic)
    • SQL as standard language
  • 1990s: Object-Oriented Model
    • Integrates database capabilities with OOP
    • Complex data types, inheritance, methods
    • Examples: ObjectStore, Versant
  • 2000s: Object-Relational Model
    • Extends relational model with OO features
    • Examples: PostgreSQL, Oracle
    • Supports user-defined types, inheritance, methods
  • 2010s-Present: NoSQL Models
    • Key-value, Document, Column-family, Graph
    • Designed for scalability and flexibility

Database Schemas vs. Database Instances

Database Schema

The intension or description of the database

  • Specified during database design
  • Relatively stable (changes infrequently)
  • Three Levels:
    1. External Schema: User views (multiple)
    2. Conceptual Schema: Complete description of database (single)
    3. Internal Schema: Physical storage structure (single)
Example of Schema Definition:
CREATE TABLE Student (
    StudentID INT PRIMARY KEY,
    Name VARCHAR(50) NOT NULL,
    Major VARCHAR(30),
    GPA DECIMAL(3,2)
);

Database Instance (State)

The extension or actual data in the database at a particular moment

  • Changes frequently with insertions, deletions, updates
  • Also called the current state or snapshot
Example of Instance:
Student Table:
StudentID | Name | Major | GPA
----------|------------|------------|-----
101 | John Doe | Computer | 3.8
102 | Jane Smith | Mathematics| 3.9
103 | Bob Wilson | Physics | 3.5

Schema Evolution

Process of changing the schema over time

  • Challenges:
    • Maintaining data consistency
    • Updating applications
    • Migrating existing data
  • Common Changes:
    • Adding new attributes/tables
    • Removing obsolete elements
    • Changing data types
    • Adding constraints

Three-Schema Architecture (ANSI/SPARC Architecture)

Proposed in 1975 to achieve data independence, this architecture separates the user applications from the physical database.

Three-Schema Architecture

The Three Levels:

  1. External Level (View Level)
    • What users see
    • Multiple external schemas (user views)
    • Each view describes part of the database relevant to a particular user group
    • Hides irrelevant data and complex data relationships
  2. Conceptual Level (Logical Level)
    • What the database contains
    • Single conceptual schema
    • Describes all entities, attributes, relationships, constraints
    • Independent of physical storage considerations
    • Serves as intermediary between internal and external levels
  3. Internal Level (Physical Level)
    • How data is stored
    • Single internal schema
    • Describes physical storage structures, access paths, file organizations
    • Uses physical data model

Mappings Between Levels:

  • External/Conceptual Mapping: Translates between external views and conceptual schema
  • Conceptual/Internal Mapping: Translates between conceptual schema and internal storage structures

Data Independence: The Key Benefit

Data Independence is the ability to change the schema at one level without affecting the schema at the next higher level.

1. Logical Data Independence

  • Ability to change the conceptual schema without having to change external schemas or application programs
  • Changes transparent to users
  • Examples of Changes Allowed:
    • Adding new entities, attributes, relationships
    • Changing constraints
    • Splitting/combining entities (if mappings adjusted)
  • Limitations: Cannot change information content that users depend on
Example of Logical Data Independence:
-- Original conceptual schema
Student(StudentID, Name, Address)

-- Modified conceptual schema (adding new attribute)
Student(StudentID, Name, Address, PhoneNumber)

-- External views and applications using only StudentID and Name
-- continue to work without modification

2. Physical Data Independence

  • Ability to change the internal schema without having to change the conceptual schema
  • Changes transparent to users and applications
  • Examples of Changes Allowed:
    • Changing file organization (sequential to indexed)
    • Using different storage structures (B-tree to hash)
    • Modifying access methods
    • Changing storage devices
    • Adding indexes for performance
Example of Physical Data Independence:
-- Changing from heap file to B+ tree organization
-- Changing block size from 2KB to 4KB
-- Adding a secondary index on Name field
-- All these changes don't affect the conceptual schema

Why Data Independence Matters:

  1. Application Durability: Applications survive changes in storage and logical organization
  2. Performance Tuning: DBAs can optimize physical storage without affecting users
  3. Evolution: Database can grow and change over time
  4. Multi-user Support: Different users can have customized views

Database Languages and Interfaces

Database Languages

  1. Data Definition Language (DDL)
    • Used to define the database schema
    • Commands: CREATE, ALTER, DROP, TRUNCATE, RENAME
    • DDL Compiler: Processes DDL statements and stores metadata in data dictionary
  2. Data Manipulation Language (DML)
    • Used to manipulate data in the database
    • Two Types:
      • Procedural DML: Specify what data is needed AND how to get it
      • Declarative DML (Non-procedural): Specify only what data is needed
    • Commands: SELECT, INSERT, UPDATE, DELETE
  3. Data Control Language (DCL)
    • Controls access to data
    • Commands: GRANT, REVOKE
  4. Transaction Control Language (TCL)
    • Manages transactions
    • Commands: COMMIT, ROLLBACK, SAVEPOINT
Example DDL:
CREATE TABLE Department (
    DeptID INT PRIMARY KEY,
    DeptName VARCHAR(50) UNIQUE,
    Location VARCHAR(100)
);

ALTER TABLE Student ADD COLUMN Email VARCHAR(100);
DROP TABLE OldTable;
Example DML:
INSERT INTO Student VALUES (104, 'Alice Brown', 'Biology', 3.7);
SELECT Name, GPA FROM Student WHERE GPA > 3.5;
UPDATE Student SET GPA = 3.9 WHERE StudentID = 101;
DELETE FROM Student WHERE StudentID = 103;
Example DCL:
GRANT SELECT, INSERT ON Student TO User1;
REVOKE DELETE ON Student FROM User2;
Example TCL:
BEGIN TRANSACTION;
UPDATE Account SET Balance = Balance - 100 WHERE AccountNo = 101;
UPDATE Account SET Balance = Balance + 100 WHERE AccountNo = 102;
COMMIT;

Storage Definition Language (SDL) and View Definition Language (VDL)

  • SDL: Specifies internal schema (usually part of DDL in modern DBMS)
  • VDL: Specifies user views (usually part of DDL using CREATE VIEW)

Database Interfaces

  1. Menu-Based Interfaces
    • Present users with lists of options (menus)
    • Suitable for naive users
    • Example: ATM interfaces, library catalog systems
  2. Form-Based Interfaces
    • Display a form for each user
    • Users fill in form fields
    • Example: Web forms, customer registration forms
  3. Graphical User Interfaces (GUI)
    • Display schema as diagrams
    • Point-and-click interaction
    • Example: phpMyAdmin, SQL Server Management Studio
  4. Natural Language Interfaces
    • Accept requests in natural language (English)
    • Parse and interpret language constructs
    • Challenges: Ambiguity, limited vocabulary
  5. Speech Input and Output
    • Voice-based interaction
    • Emerging technology with AI integration
  6. Interfaces for Parametric Users
    • Function keys for frequent operations
    • Used by bank tellers, reservation clerks
  7. Interfaces for Database Administrators (DBA)
    • Privileged commands for system control
    • Performance monitoring tools
    • Backup and recovery utilities
  8. Application Programming Interfaces (APIs)
    • Allow programs to interact with DBMS
    • Examples: ODBC, JDBC, ADO.NET, Hibernate

Database System Architectures

1. Centralized Database Systems

  • All DBMS functionality, application programs, and user interface processing on single machine
  • Advantages:
    • Simple to manage and maintain
    • Data consistency easy to ensure
    • Lower initial cost for small systems
  • Disadvantages:
    • Single point of failure
    • Limited scalability
    • Performance bottlenecks
  • Types:
    • Teleprocessing: Dumb terminals connected to central computer
    • Single-user: Desktop database systems (Microsoft Access)

2. Client-Server Architecture

  • Separates functions between clients and servers
  • Client: Presentation layer + application logic
  • Server: Database storage + processing
Two-Tier Architecture:
Two-Tier Architecture
  • Thin Client: Only presentation (most processing on server)
  • Thick Client: Both presentation and business logic
  • Advantages: Better scalability than centralized
  • Disadvantages: Business logic distribution issues, maintenance challenges
Three-Tier Architecture:
Three-Tier Architecture
  • Advantages:
    • Better security (client doesn't access database directly)
    • Thin clients (easier deployment)
    • Business logic centralized
    • Improved scalability
  • Common in: Web applications, enterprise systems
N-Tier Architecture:
  • Extension of three-tier with additional layers
  • Example: Web browser → Web server → Application server → Database server

3. Parallel Database Systems

  • Multiple processors and disks for parallel processing
  • Architectures:
    • Shared Memory: Processors share main memory
    • Shared Disk: Processors have private memory but share disks
    • Shared Nothing: Each processor has private memory and disks
  • Used in data warehousing, big data processing

4. Distributed Database Systems

  • Data distributed across multiple sites connected by network
  • Types:
    • Homogeneous: Same DBMS at all sites
    • Heterogeneous: Different DBMS at different sites
  • Advantages: Local autonomy, improved availability, scalability
  • Challenges: Distributed query processing, concurrency control, recovery

1.2 CONCEPTUAL DATA MODELING USING ENTITY-RELATIONSHIP (ER) MODEL

Introduction to ER Modeling

The Entity-Relationship (ER) Model is a high-level conceptual data model developed by Peter Chen in 1976. It provides a graphical representation (ER diagrams) for database design.

Purpose: To model the data aspects of a business domain in a way that is understandable to both technical and non-technical stakeholders.

Basic Concepts of ER Model

1. Entities

  • An entity is a "thing" in the real world with an independent existence
  • Can be physical (person, car, house) or conceptual (company, job, course)
Entity Type:
  • A collection (set) of entities that have the same attributes
  • Example: STUDENT, COURSE, EMPLOYEE
  • Represented by rectangle in ER diagrams
Entity Set:
  • Collection of all entities of a particular entity type at any point in time
  • Example: All students currently enrolled in a university
Entity Instance:
  • A single occurrence of an entity type
  • Example: Student with ID 101 named "John Doe"

2. Attributes

  • Properties or characteristics of entities
  • Represented by ovals connected to entity rectangles
Types of Attributes:
  1. Simple (Atomic) vs. Composite Attributes
    • Simple: Cannot be divided further (e.g., Age, Salary)
    • Composite: Can be divided into sub-parts (e.g., Name → FirstName, LastName)
  2. Single-valued vs. Multi-valued Attributes
    • Single-valued: Has single value for each entity (e.g., DateOfBirth)
    • Multi-valued: May have multiple values (e.g., PhoneNumbers, EmailAddresses)
    • Represented by double oval
  3. Stored vs. Derived Attributes
    • Stored: Physically stored in database (e.g., DateOfBirth)
    • Derived: Calculated from other attributes (e.g., Age from DateOfBirth)
    • Represented by dashed oval
  4. Complex Attributes
    • Combination of composite and multi-valued
    • Example: Address with multiple phone numbers
  5. NULL-valued Attributes
    • Value not known or not applicable
    • Types of NULL:
      • Unknown: Value exists but not known
      • Not applicable: Attribute doesn't apply to this entity
      • Missing: Value exists but not recorded
Types of Attributes in ER Model

3. Keys

Identifiers that uniquely determine entities in entity sets.

  1. Superkey
    • Set of one or more attributes that uniquely identifies an entity
    • Example for STUDENT: {StudentID}, {StudentID, Name}, {SSN}, etc.
  2. Candidate Key
    • Minimal superkey (no proper subset is a superkey)
    • Entity may have multiple candidate keys
    • Properties:
      • Uniqueness: Each entity has distinct value
      • Minimality: No extraneous attributes
    • Example for STUDENT: StudentID, SSN (if both are available)
  3. Primary Key
    • Candidate key chosen by database designer as main identifier
    • Represented by underlining in ER diagrams
    • Selection Criteria:
      • Minimal (fewest attributes)
      • Stable (values don't change)
      • Simple (easy to use and remember)
    • Example: Choosing StudentID over SSN as primary key
  4. Secondary Key
    • Attribute used for retrieval but not necessarily unique
    • Example: LastName for searching students
  5. Foreign Key (in relational model, implied in ER)
    • Attribute(s) in one relation that reference primary key of another relation

4. Value Sets (Domains)

  • Set of allowable values for an attribute
  • Specification includes:
    • Data type (integer, string, date, etc.)
    • Format/length constraints
    • Range or list of allowable values
  • Example: GPA domain: decimal between 0.0 and 4.0

Relationship Types and Sets

1. Relationship Type

  • Association among two or more entity types
  • Represented by diamond in ER diagrams
  • Example: ENROLLS_IN between STUDENT and COURSE

2. Relationship Set

  • Collection of relationships of the same type
  • Mathematical Definition: Subset of Cartesian product of participating entity sets
  • For binary relationship between entity sets E1 and E2:
    • Relationship set R ⊆ {(e1, e2) | e1 ∈ E1, e2 ∈ E2}
  • Example: All (student, course) pairs where student is enrolled in course

3. Degree of Relationship

Number of participating entity types in a relationship.

  1. Unary (Recursive) Relationship
    • Relationship between entities of same entity type
    • Example:
      • EMPLOYEE supervises EMPLOYEE
      • COURSE is_prerequisite_for COURSE
  2. Binary Relationship (Most Common)
    • Relationship between two entity types
    • Example: STUDENT takes COURSE
  3. Ternary Relationship
    • Relationship among three entity types
    • Example: DOCTOR prescribes DRUG for PATIENT
  4. N-ary Relationship
    • Relationship among n entity types (n > 2)
    • Important Note: Not equivalent to multiple binary relationships
    • Example: SUPPLIER supplies PART to PROJECT
Relationship Types: Unary, Binary, and Ternary

4. Roles in Relationships

  • Function that an entity plays in a relationship
  • Important when same entity type participates multiple times
  • Example: In EMPLOYEE supervises EMPLOYEE:
    • Roles: supervisor (one side), subordinate (other side)
  • Represented by labeling lines in ER diagrams

5. Attributes of Relationships

  • Relationships can have attributes
  • Example: ENROLLS_IN relationship between STUDENT and COURSE can have:
    • Semester
    • Grade
    • EnrollmentDate
Relationship with Attributes

Structural Constraints on Relationships

1. Cardinality Ratio Constraints

Specify the number of relationship instances an entity can participate in.

Four Types for Binary Relationships:
  1. One-to-One (1:1)
    • Each entity in E1 associated with at most one entity in E2
    • Each entity in E2 associated with at most one entity in E1
    • Example: EMPLOYEE manages DEPARTMENT
      • Each employee manages at most one department
      • Each department has at most one manager
    • Notation: (1:1) or numbers on relationship lines
  2. One-to-Many (1:N)
    • Each entity in E1 associated with many entities in E2
    • Each entity in E2 associated with at most one entity in E1
    • Example: DEPARTMENT employs EMPLOYEE
      • Each department employs many employees
      • Each employee works in at most one department
    • Notation: (1:N)
  3. Many-to-One (N:1)
    • Inverse of 1:N
    • Example: EMPLOYEE works_in DEPARTMENT (N:1)
  4. Many-to-Many (M:N)
    • Each entity in E1 associated with many entities in E2
    • Each entity in E2 associated with many
