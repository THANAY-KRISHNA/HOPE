CAP Theorem

Brewer's Conjecture (2000): A distributed data store can provide only two of the following three guarantees:
1. Consistency (C): Every read receives the most recent write or an error
2. Availability (A): Every request receives a (non-error) response
3. Partition Tolerance (P): System continues despite network partitions
Figure: CAP Theorem Triangle - The Trade-offs

From ACID to BASE: Relaxed consistency model for distributed systems.

Column-Family Stores (Wide-Column Stores)

Definition: Distributed, scalable tabular databases with flexible columns per row.
Figure: Column-Family Store vs RDBMS Structure

Cassandra Query Language (CQL) Example

-- Create table with partition and clustering keys
                CREATE TABLE sensor_data (
                sensor_id UUID,
                year_month TEXT, -- Partition by month
                timestamp TIMESTAMP,
                temperature DOUBLE,
                humidity DOUBLE,
                pressure DOUBLE,
                PRIMARY KEY ((sensor_id, year_month), timestamp)
                ) WITH CLUSTERING ORDER BY (timestamp DESC);

                -- Query for last 24 hours of data
                SELECT * FROM sensor_data
                WHERE sensor_id = ?
                AND year_month = '2024-01'
                AND timestamp >= '2024-01-15 00:00:00'
                ORDER BY timestamp DESC;

Graph Databases

Definition: Optimized for storing and querying connected data (nodes and relationships).
Figure: Graph Database Structure - Social Network Example

Cypher Query Examples

-- Create nodes and relationships
                CREATE (alice:Person {name: 'Alice', age: 30})
                CREATE (bob:Person {name: 'Bob', age: 28})
                CREATE (python:Skill {name: 'Python'})
                CREATE (alice)-[:FRIEND]->(bob)
                CREATE (alice)-[:KNOWS {since: 2019}]->(python)

                -- Find connections between Alice and Bob
                MATCH path = shortestPath((alice:Person {name: 'Alice'})-[*]-(bob:Person {name: 'Bob'}))
                RETURN path, length(path)

                -- Recommendation: People Alice may know
                MATCH (alice:Person {name: 'Alice'})-[:FRIEND]->(friend)-[:FRIEND]->(friendOfFriend)
                WHERE NOT (alice)-[:FRIEND]->(friendOfFriend)
                AND alice <> friendOfFriend
                    RETURN friendOfFriend.name, COUNT(*) as mutualFriends
                    ORDER BY mutualFriends DESC

Module 4 Summary

Key Concepts to Remember

Fundamental Concepts:
1. CAP Theorem: Can only guarantee 2 of 3: Consistency, Availability, Partition Tolerance
2. BASE Properties: Basically Available, Soft State, Eventually Consistent
3. Schema-on-Read vs Schema-on-Write: Flexibility vs structure
NoSQL Categories:
1. Key-Value Stores: Simple hash tables (Redis, DynamoDB)
2. Document Stores: JSON/BSON documents (MongoDB, CouchDB)
3. Column-Family Stores: Sparse, wide tables (Cassandra, HBase)
4. Graph Databases: Nodes and relationships (Neo4j, Amazon Neptune)

Use Case Guidelines

Database Type When to Use Examples
Key-Value Stores Need simple key-based access, high-performance caching, session management Redis, DynamoDB, Memcached
Document Stores Data is naturally hierarchical, schema evolves frequently, content management MongoDB, CouchDB, Couchbase
Column-Family Stores High write throughput needed, time-series data, horizontal scalability Cassandra, HBase, ScyllaDB
Graph Databases Relationships are primary focus, social networks, fraud detection Neo4j, Amazon Neptune, JanusGraph
Relational Databases Complex transactions required, ad-hoc querying, mature stable schema PostgreSQL, MySQL, Oracle

Final Thought

NoSQL is not a replacement for RDBMS, but a complementary set of tools.

The modern data landscape requires understanding both paradigms and choosing the right tool for each job. Most large-scale applications today use a combination of SQL and NoSQL databases in a polyglot persistence architecture.

Remember: "To a man with a hammer, everything looks like a nail." Don't force NoSQL where it doesn't fit, and don't cling to RDBMS when NoSQL offers clear advantages. Understand the trade-offs and make informed decisions based on your specific requirements.
